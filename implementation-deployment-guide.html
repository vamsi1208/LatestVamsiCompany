Agents Platform Guide

Everything you need to understand implementation, runtime flow, and deployment for the Agents Platform project.

1. Architecture Overview

This platform delivers a React frontend, Cognito authentication, API Gateway, multiple Lambda agents, Bedrock model calls, and DynamoDB session storage.

Key components
  • CloudFront + S3 frontend: serves the React single-page app.
  • Cognito User Pool: handles sign-up/login and JWT issuance.
  • API Gateway: exposes authenticated REST endpoints under /agents/*.
  • Lambda agents:
    • test-case-agent
    • playwright-agent
    • history-agent
  • Bedrock: model inference service used by the first two agents.
  • DynamoDB: stores session history for agent calls.
Request flow
React UI -> CloudFront -> API Gateway -> Lambda -> Bedrock/DynamoDB -> API Gateway -> React UI
2. Frontend Implementation
Frontend structure

Located in agents-platform/frontend. It is a React + Vite app with Amplify configuration and Axios for API requests.

Auth setup

src/lib/amplify.ts initializes Cognito configuration using Vite env variables:

import { Amplify } from 'aws-amplify';

Amplify.configure({
  Auth: {
    Cognito: {
      userPoolId: import.meta.env.VITE_USER_POOL_ID,
      userPoolClientId: import.meta.env.VITE_USER_POOL_CLIENT_ID,
    },
  },
});
API client

src/lib/api.ts creates an Axios client with automatic Cognito JWT injection:

import axios from 'axios';
import { fetchAuthSession } from 'aws-amplify/auth';

const api = axios.create({ baseURL: import.meta.env.VITE_API_URL });

api.interceptors.request.use(async (config) => {
  const session = await fetchAuthSession();
  const token = session.tokens?.idToken?.toString();
  if (token) config.headers.Authorization = token;
  return config;
});

export default api;
Agent page UI

src/components/AgentPage.tsx renders the input box, submit flow, and markdown output for any agent.

const res = await api.post(endpoint, { input });
setOutput(res.data.output);
3. Backend Lambda Agents
Common behavior

All Lambda agent handlers parse JSON input, call AWS Bedrock or DynamoDB, store session history, and return JSON.

The API Gateway authorizer injects Cognito claims into event.requestContext.authorizer.claims. The Lambda uses that to set userId.
Test case agent

Implemented in backend/agents/test-case-agent/handler.js. It:

  • reads event.body.input
  • invokes Bedrock with a QA prompt
  • stores results in DynamoDB
  • returns { sessionId, output }
const response = await bedrock.send(new ConverseCommand({
  modelId: process.env.BEDROCK_MODEL_ID,
  system: [{ text: SYSTEM_PROMPT }],
  messages: [{ role: 'user', content: [{ text: input }] }],
  inferenceConfig: { maxTokens: 2048 },
}));
Playwright agent

In backend/agents/playwright-agent/handler.js. It behaves similarly but prompts for a runnable Playwright script instead of test cases.

History agent

In backend/agents/history-agent/handler.js. It queries the DynamoDB table by userId and returns the last 20 sessions.

Deployment steps
  1. Open terminal in agents-platform/infrastructure.
  2. Run npm install if not installed.
  3. Run npm run build to compile the CDK TypeScript stack.
  4. Run npx cdk deploy (or npm run cdk deploy).
  5. Note CDK outputs: CloudFrontURL, ApiGatewayURL, UserPoolId, UserPoolClientId, FrontendBucketName.
  6. Update frontend env variables in frontend/.env.local or Vite environment with those outputs.
  7. Build frontend: cd ../frontend && npm install && npm run build.
  8. Deploy static files: aws s3 sync dist/ s3://<bucket-name>.
  9. Open the CloudFront URL in a browser.
5. Testing and Troubleshooting
Local test script

agents-platform/test-api.js shows how the system is exercised using Cognito and API Gateway.

const USER_POOL_ID = 'us-east-1_Aoxgqntxz';
const CLIENT_ID = '5b7kb05lkkti1s91lb8o1n3im5';
const API_URL = 'https://a3i7g4cl61.execute-api.us-east-1.amazonaws.com/prod';

It authenticates via Cognito and calls /agents/test-case.

Common failure points
  • Cognito auth failure: invalid credentials or wrong client config.
  • API Gateway 500: Lambda error, usually from Bedrock failures or invalid input parsing.
  • Bad env vars: Lambda expects TABLE_NAME, BEDROCK_MODEL_ID, BEDROCK_REGION.
  • Permissions: Lambda needs Bedrock and DynamoDB access in its IAM role.
Debugging tips
  • Check CloudWatch logs for the failing Lambda.
  • Confirm API Gateway method request and integration settings.
  • Validate the Authorization header is the Cognito ID token, not an access token.
  • Inspect CloudFront/S3 access if the frontend does not load.
6. Environment variables and config
Lambda env vars
const lambdaEnv = {
  TABLE_NAME: sessionsTable.tableName,
  BEDROCK_MODEL_ID: 'us.amazon.nova-lite-v1:0',
  BEDROCK_REGION: 'us-west-2',
};
Frontend env vars
  • VITE_API_URL: API Gateway base URL
  • VITE_USER_POOL_ID: Cognito User Pool ID
  • VITE_USER_POOL_CLIENT_ID: Cognito App Client ID
The frontend reads these from Vite env, and the API client adds Authorization automatically when signed in.
